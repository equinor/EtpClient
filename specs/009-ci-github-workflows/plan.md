# Implementation Plan: CI GitHub Actions Workflows

**Branch**: `009-ci-github-workflows` | **Date**: 2026-04-14 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/009-ci-github-workflows/spec.md`

**Path Rule**: Use repository-relative paths only. Never include machine-local absolute filesystem paths in generated artifacts.

## Summary

Add GitHub Actions CI to the repository. A single `ci.yml` workflow triggers on pull requests and pushes to `main`, runs path-filtered build-and-test jobs per component (library, SampleConsole, EtpExplorer), and surfaces a status badge in `README.md`. A separate `publish-nuget.yml` workflow provides a manual `workflow_dispatch` trigger for publishing `EtpClient` as a NuGet package to the GitHub Package Repository. `EtpClient.csproj` receives the minimum NuGet metadata required to produce a well-formed package.

## Technical Context

**Language/Version**: YAML (GitHub Actions), C# / .NET 10  
**Primary Dependencies**: GitHub Actions built-in (`actions/checkout@v4`, `actions/setup-dotnet@v4`); `dorny/paths-filter@v3` for path-change detection (see research.md)  
**Storage**: N/A  
**Testing**: xUnit via `dotnet test` — four existing test projects run across three CI jobs  
**Target Platform**: `ubuntu-latest` GitHub-hosted runner  
**Project Type**: CI / infrastructure configuration + minor library metadata addition  
**Performance Goals**: Full CI feedback (build + tests) under 5 minutes for a typical PR  
**Constraints**: No external secrets beyond `GITHUB_TOKEN`; no external ETP server; path-filtered so unrelated components do not rebuild  
**Scale/Scope**: 3 independently buildable components; 2 workflow YAML files; 1 minor `.csproj` metadata change; 1 README badge
